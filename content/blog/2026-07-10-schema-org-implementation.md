# 構造化データ（Schema.org）の実装でSEOと検索表示を強化する方法

Googleの検索結果を眺めていると、レビューの星マークや営業時間、FAQがアコーディオン形式で表示されているリッチリザルトを見かけることが増えてきました。あれを実現しているのが「構造化データ」です。

中小企業のWebサイトでも適切に実装すれば、検索結果での見た目が変わり、クリック率の向上につながる可能性があります。ただし「実装すれば順位が上がる」という単純な話ではなく、正しく理解して使うことが大切です。この記事では、構造化データの基本から具体的な実装手順、実務で使える注意点まで解説します。

---

## 構造化データとは何か、なぜ今重要なのか

構造化データとは、Webページの内容を検索エンジンが機械的に理解しやすい形式で記述したコードのことです。HTMLだけでは「このテキストが店名なのか商品名なのか」を検索エンジンが判断しにくい場面があります。構造化データを使うことで「これは店舗名です」「これは評価点数です」と明示的に伝えられます。

標準的な語彙として使われているのが**Schema.org**です。Google・Microsoft・Yahoo・Yandexが共同で策定した仕様で、ここで定義されたプロパティ名を使うことで、主要な検索エンジン全般に情報を正確に伝えられます。

構造化データが注目される理由は大きく2つです。

1. **リッチリザルトの表示資格を得られる**  
   星評価・価格・FAQ・パンくずリストなど、通常の青いリンクとは異なる表示形式（リッチリザルト）が表示される可能性が生まれます。これにより検索結果上での視認性が高まります。

2. **AIO（AI Overviews）への対応**  
   GoogleのAI生成サマリーが普及しつつある中、構造化データによって整理されたコンテンツはAIが参照しやすくなります。今後の検索体験の変化を見据えても、実装しておく意義があります。

---

## 実装形式はJSON-LDを選ぶのが基本

構造化データの記述形式には「JSON-LD」「Microdata」「RDFa」の3種類がありますが、**Googleが推奨しているのはJSON-LD**です。

JSON-LDの最大の利点は、HTMLの本文とは分離してページのどこにでも記述できる点です。MicrodataやRDFaのようにHTMLタグの属性として埋め込む必要がないため、既存のページデザインに影響を与えずに追加できます。WordPressなら`<head>`内やフッターに`<script type="application/ld+json">`タグとして差し込む形が一般的です。

以下は店舗情報（LocalBusiness）の基本的なJSON-LDの例です。

```json
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "LocalBusiness",
  "name": "株式会社サイプレス",
  "url": "https://example.com",
  "telephone": "03-XXXX-XXXX",
  "address": {
    "@type": "PostalAddress",
    "streetAddress": "〇〇区〇〇1-2-3",
    "addressLocality": "東京都",
    "postalCode": "100-0000",
    "addressCountry": "JP"
  },
  "openingHours": "Mo-Fr 09:00-18:00"
}
</script>
```

記述した後は、Googleの**リッチリザルトテスト**（search.google.com/test/rich-results）でエラーが出ていないか必ず確認してください。

## 業種・目的別に使いたいスキーマタイプ

Schema.orgには800以上のタイプが定義されていますが、中小企業のサイトで実用性が高いのは以下のタイプです。

### LocalBusiness（店舗・事務所）
MEO対策とも連動しやすく、Googleビジネスプロフィールと合わせて整備することで、ローカル検索での情報一致性が高まります。業種に応じて`Restaurant`・`MedicalBusiness`・`LegalService`などのサブタイプを使い分けると、より詳細な情報を伝えられます。

### FAQPage（よくある質問）
FAQ形式のページに実装すると、検索結果でアコーディオン表示が出ることがあります。サービスページや料金ページの下部によくある質問を設置しているなら、積極的に実装を検討する価値があります。

```json
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [{
    "@type": "Question",
    "name": "ホームページ制作の納期はどのくらいですか？",
    "acceptedAnswer": {
      "@type": "Answer",
      "text": "規模によって異なりますが、標準的なコーポレートサイトで1〜2ヶ月程度が目安です。"
    }
  }]
}
```

### Article・BlogPosting（ブログ・コラム）
コンテンツの作成日・更新日・著者情報を明示することで、検索エンジンがコンテンツの鮮度と信頼性を判断しやすくなります。E-E-A-T（経験・専門性・権威性・信頼性）の観点でも、著者情報の構造化は意識しておきたいポイントです。

### BreadcrumbList（パンくずリスト）
サイト構造を検索エンジンに伝えるとともに、検索結果のURLの代わりにパンくずが表示されることがあります。ECサイトや階層の深いコンテンツサイトで特に効果的です。

## 実装時によくある失敗と注意点

構造化データは「入れれば完璧」というものではなく、誤った実装はGoogleのガイドライン違反になる場合もあります。

**ページの内容と一致しない情報を記述しない**  
実際のページに表示されていない評価点数や営業時間を構造化データに記述することは、スパムとみなされる可能性があります。構造化データの内容は、ページ上に実際に表示されている内容と一致させることが原則です。

**すべてのページに同じ構造化データを貼らない**  
トップページ・サービスページ・ブログ記事では適切なスキーマタイプが異なります。サイト全体に画一的なコードを挿入すると、かえって誤った情報を伝えることになります。

**Search Consoleの「拡張」レポートを定期確認する**  
実装後はGoogle Search Consoleの「拡張」メニューから構造化データのエラーや警告を確認できます。「有効（警告あり）」の状態はリッチリザルトが表示される可能性がありますが、「無効」の状態ではエラーの原因を特定して修正が必要です。

**プラグインに頼りすぎない**  
WordPressの構造化データプラグインは便利ですが、自動生成されたコードが常に正確とは限りません。特にLocalBusinessのプロパティは手動で確認・修正することをお勧めします。

## 実装の優先順位の考え方

すべてを一度に実装しようとすると作業が分散します。以下の順序で取り組むと効率的です。

1. **Organization / LocalBusiness** → サイト全体・企業情報として最優先
2. **BreadcrumbList** → サイト構造が明確なら早めに
3. **FAQPage** → 既にFAQコンテンツがあれば即実装可能
4. **Article / BlogPosting** → ブログ更新頻度が高いサイトで効果的
5. **Product / Review** → ECサイト・商品紹介ページに

リッチリザルトが表示されるかどうかはGoogleが判断するため、実装したからといって必ず表示されるわけではありません。ただし、表示される「資格」を持つためには実装していることが前提条件です。

---

## まとめ

構造化データは、検索エンジンとの「コミュニケーション精度」を上げる手段です。SEO対策の中でも比較的見落とされがちですが、競合が実装していない状況であれば、リッチリザルトで差別化できるチャンスでもあります。

まず自社サイトの現状をGoogle Search Consoleのリッチリザルトテストで確認し、LocalBusinessとFAQPageから着手してみることをお勧めします。

構造化データの実装はコーディングの知識が必要で、誤った実装はかえって逆効果になる場合もあります。「自社でやってみたがエラーが解決しない」「そもそも何を実装すべきかわからない」という場合は、株式会社サイプレスにご相談ください。SEO対策・ホームページ制作の観点から、サイトの状況に合った構造化データの整備をご提案しています。
